입력:

raw_text – 영업이 업로드한 원본 문서(규칙/약정/안내 등)

출력:
다음과 같은 구조의 JSON 객체를 반환하십시오. 섹션 수는 문서 내용을 기반으로 AI가 적절히 결정하며, 4~8개 범위 내에서 논리적으로 묶어주세요.

{
  "sections": [
    {
      "name": "섹션 이름",      // 예: "운영시간/휴무", "환불/정산", "예약/변경", "강좌/정원/대기" 등
      "content": "문서에서 추출·정제한 내용. 문장은 명확하고 간결하게, 필요한 경우 bullet point 사용",
      "issues": [
        {
          "type": "MISSING|AMBIGUOUS|CONFLICT|PII_RISK|API_NEEDED",
          "message": "해당 섹션에서 발견한 문제 설명",
          "suggestion": "수정·보완을 위한 제안(예: 추가 정보 요청 또는 모호 표현 수정 안내)"
        }
      ]
    },
    ...
  ],
  "clarification_questions": [
    "문서 전반에서 빠진 핵심 정보나 모호한 사항에 대해 작성자가 답해야 할 질문"
  ],
  "change_summary": "AI가 수행한 주요 정리·보완 작업 요약"
}

지침:

섹션 분류: 문서 내용을 분석하여 자연스럽게 묶이는 주제별로 4~8개 섹션을 생성하고, 각 섹션에 적합한 이름(name)을 지정합니다. 형식이 고정되어 있지 않으므로 문서 특성에 맞게 자유롭게 분류하되, 같은 주제가 분산되지 않게 유의하세요.

내용 정제: 각 섹션의 content에는 원본 내용의 의미를 유지한 채 더 명확하고 간결한 표현으로 정리합니다. 긴 문장은 짧게 나누고, 필요할 때 bullet point(띄어쓰기 번호, 하이픈 등)로 나열해 가독성을 높입니다.

이슈 탐지: 섹션마다 issues 배열을 작성합니다.

MISSING: 중요한 정보가 누락된 경우(예: 환불 기한 명시가 없음).

AMBIGUOUS: “상황에 따라”, “적당히” 등 모호한 표현을 사용한 경우.

CONFLICT: 서로 상충되는 규정이 발견된 경우.

PII_RISK: 주민번호, 전화번호 등 개인 정보를 노출하는 경우.

API_NEEDED: 실제 시스템 액션이 필요해 보이는 문구(예약 생성, 문자 발송 등)가 있을 경우.
각 이슈에 대해 무엇이 문제인지(message)와 이를 어떻게 해결하면 좋을지(suggestion) 한 줄로 적습니다.

추가 질문 작성: clarification_questions에는 문서 전체에서 이해를 위해 필요한 추가 질문을 나열합니다. 예를 들어, “예약 취소 가능 기한을 명확히 알려주세요”, “환불 수수료 부과 기준을 상세히 설명해 주세요”처럼 구체적으로 작성합니다. 작성자가 답변하면 그 내용을 반영하여 최종 매뉴얼을 완성할 수 있도록 유도합니다.

요약: change_summary에는 AI가 수행한 주요 보정 사항을 요약합니다. 예를 들어, “모호 표현 3건 수정, 누락 정보 2건 발견” 등.

정보 유지: 문서에 없는 정보를 새로 만들어 넣지 마세요. 필요한 정보가 없을 경우, 이를 clarification_questions나 issues에 명확히 요청합니다.
